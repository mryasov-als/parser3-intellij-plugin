# Область видимости в Parser3

## Введение

Parser3 — это серверный язык программирования с особой моделью области видимости, основанной на классах и методах. Понимание этой модели критически важно для корректной работы автодополнения и навигации в IDE.

---

## 1. Структура файла Parser3

Типичный файл `.p` может содержать:

```parser3
# Код вне классов — это "MAIN" класс (неявный)
@main[]
^connect[mysql://...]

# Явно объявленный класс
@CLASS
User

@BASE
BaseUser

@create[name;email]
$self.name[$name]
$self.email[$email]

@validate[]
^if(!$self.email){
    ^throw[error;Email required]
}

# Ещё один класс в том же файле
@CLASS
Admin

@BASE
User

@ban[user_id]
...
```

---

## 2. Типы классов

### 2.1. MAIN класс (неявный)

**MAIN** — это код, находящийся вне директив `@CLASS`. Каждый файл имеет неявный MAIN класс.

```parser3
# Это MAIN класс
@main[]
^process[]

@process[]
$result[...]
```

**Особенности MAIN:**
- Методы MAIN доступны из любого места через `^MAIN:method[]`
- MAIN не может иметь `@BASE`
- `^self.method[]` в MAIN эквивалентен `^MAIN:method[]` — ищет во всех видимых MAIN методах

### 2.2. Именованные классы (@CLASS)

Классы объявляются через директиву `@CLASS`:

```parser3
@CLASS
ClassName
```

**Особенности:**
- Могут наследовать от других классов через `@BASE`
- Имеют собственное пространство имён для методов
- Могут быть созданы как объекты: `$obj[^ClassName::create[]]`
- Имя класса может начинаться с цифры: `@CLASS 222` — это валидный идентификатор класса

### 2.3. Partial-классы (@OPTIONS partial)

Класс может быть объявлен в нескольких файлах с помощью директивы `@OPTIONS partial`:

```parser3
# User.p
@CLASS
User

@OPTIONS
partial

@create[name]
$self.name[$name]

# UserValidation.p
@CLASS
User

@OPTIONS
partial

@validate[]
^if(!$self.name){^throw[error]}
```

**Особенности partial-классов:**
- Один класс может быть определён в нескольких файлах
- Все файлы должны иметь `@OPTIONS partial`
- Методы из всех файлов объединяются в один класс
- При навигации IDE должна искать методы во всех файлах partial-класса

## 3. Способы вызова методов

### 3.1. Три типа скобок

Parser3 поддерживает три типа скобок для вызова методов — они полностью взаимозаменяемы:

| Скобки | Пример | Описание |
|--------|--------|----------|
| `[]` | `^method[]` | Квадратные скобки (стандартный стиль) |
| `()` | `^method()` | Круглые скобки |
| `{}` | `^method{}` | Фигурные скобки (часто для SQL и многострочного контента) |

**Примеры эквивалентных вызовов:**
```parser3
^process[]
^process()
^process{}

^MAIN:connect[]
^MAIN:connect()
^MAIN:connect{}

^self.validate[]
^self.validate()
^self.validate{}

^User::create[name;email]
^User::create(name;email)
^User::create{name;email}
```

**Использование на практике:**
- `[]` — обычные вызовы методов
- `()` — часто для условий и выражений: `^if(condition){...}`
- `{}` — часто для SQL и многострочного контента: `^void:sql{SELECT * FROM users}`

### 3.2. Простой вызов: `^method[]`

```parser3
^process[]
```

**Область поиска метода:**
1. Методы текущего класса
2. Методы базовых классов (вся цепочка `@BASE`)
3. Методы MAIN текущего файла
4. Методы MAIN из подключённых файлов (`^use[]`, `@USE`, `auto.p`)

### 3.3. Вызов через self: `^self.method[]`

```parser3
^self.validate[]
```

**Поведение зависит от контекста:**

**В именованном классе (@CLASS):**
- Ищет только в текущем классе и цепочке `@BASE`
- НЕ ищет в MAIN и других классах
- Используется для явного вызова метода текущего объекта

**В MAIN классе (вне @CLASS):**
- `^self.method[]` эквивалентен `^MAIN:method[]`
- Ищет во всех видимых MAIN методах (текущий файл + подключённые)
- `self` в MAIN — это сам MAIN

```parser3
# В MAIN классе
@main[]
^self.process[]   # Эквивалентно ^MAIN:process[]
^MAIN:process[]   # То же самое

@process[]
...

@CLASS
User

@run[]
^self.validate[]  # Только User.validate или BASE.validate
                  # НЕ ищет в MAIN
```

### 3.4. Вызов из MAIN: `^MAIN:method[]`

```parser3
^MAIN:connect[]
```

**Область поиска:**
- Только методы MAIN класса (код вне `@CLASS`)
- Из текущего файла и подключённых через `^use[]`, `@USE`, `auto.p`

**Использование:** Явный вызов глобальных функций/методов.

### 3.5. Вызов из базового класса: `^BASE:method[]`

```parser3
@CLASS
Admin

@BASE
User

@create[name;email;role]
^BASE:create[$name;$email]  # Вызов create[] из User
$self.role[$role]
```

**Область поиска:**
- Методы класса, указанного в `@BASE`
- Методы базовых классов базового класса (вся цепочка наследования)

**Использование:** Вызов переопределённого метода родительского класса (аналог `super` в других языках).

**Важно:** `^BASE:method[]` валиден только в классе с `@BASE`. В MAIN или классе без `@BASE` — это ошибка.

### 3.6. Вызов статического метода класса: `^ClassName:method[]`

```parser3
^User:validate[$email]
```

**Область поиска:**
- Статические методы указанного класса
- Методы базовых классов указанного класса

**Использование:** Вызов метода конкретного класса без создания объекта.

### 3.7. Вызов конструктора: `^ClassName::create[]`

```parser3
$user[^User::create[John;john@example.com]]
```

**Двойное двоеточие (`::`)** — создание нового объекта класса.

## 4. Наследование (@BASE)

### 4.1. Объявление наследования

```parser3
@CLASS
ChildClass

@BASE
ParentClass
```

### 4.2. Цепочка наследования

```parser3
@CLASS
BaseEntity
@create[]
...

@CLASS  
User
@BASE
BaseEntity
@create[name]
...

@CLASS
Admin
@BASE
User
@create[name;role]
^BASE:create[$name]  # Вызывает User::create
...
```

**Цепочка:** `Admin` → `User` → `BaseEntity`

### 4.3. Поиск методов в иерархии

При вызове `^self.method[]` или `^method[]` поиск идёт:
1. В текущем классе
2. В `@BASE` текущего класса
3. В `@BASE` базового класса
4. И так далее до конца цепочки

### 4.4. Вызовы методов наследников

При поиске вызовов метода (обратная навигация) нужно учитывать:
- Метод `@create[]` в классе `User` может быть вызван как `^self.create[]` из `UserEx` (наследника)
- `^BASE:create[]` из наследника — это вызов метода родителя

## 5. Подключение файлов

### 5.1. Директива @USE

```parser3
@USE
/classes/User.p
/classes/Admin.p
```

**Выполняется:** На этапе компиляции, до выполнения кода.

### 5.2. Метод ^use[]

```parser3
^use[/classes/User.p]
```

**Выполняется:** Во время выполнения кода.

### 5.3. Автоматическое подключение auto.p

Parser3 автоматически подключает файлы `auto.p` по иерархии директорий **сверху вниз** — от корня проекта к директории текущего файла.

**Пример структуры:**
```
/project/
├── auto.p                    # (1) Подключается первым
├── classes/
│   ├── auto.p                # (2) Подключается вторым
│   └── user/
│       ├── auto.p            # (3) Подключается третьим
│       └── User.p            # ← Текущий файл
```

При выполнении `/project/classes/user/User.p` автоматически подключаются:
1. `/project/auto.p`
2. `/project/classes/auto.p`
3. `/project/classes/user/auto.p`

**Порядок:** От корня проекта к директории текущего файла (сверху вниз).

**Использование auto.p:**
- Общие функции и утилиты
- Подключение зависимостей (`^use[]`)
- Инициализация (подключение к БД, конфигурация)
- Определение глобальных классов

### 5.4. Метод @auto[]

Метод `@auto[]` — это инициализатор файла. Если в файле определён метод `@auto[]`, он автоматически выполняется при первом обращении к этому файлу.

```parser3
@auto[]
# Этот код выполнится автоматически при первом обращении к файлу
^connect_to_db[]
$initialized(true)

@process[]
...
```

**Важно:** Метод `@auto[]` не влияет на подключение файла. Файл всё равно должен быть явно подключён через `@USE`, `^use[]` или быть файлом `auto.p`.

### 5.5. Метод @autouse[] — автозагрузка классов

Метод `@autouse[]` в MAIN классе позволяет автоматически загружать классы по требованию, без явного `^use[]`.

```parser3
# /project/auto.p
@autouse[name]
# Вызывается автоматически при первом обращении к классу $name
^use[/classes/$name.p]
```

**Как работает:**
- При обращении к классу, который ещё не загружен (например, `^User::create[]`), Parser3 вызывает `@autouse[]` с именем класса
- Метод `@autouse[]` должен подключить файл с нужным классом
- После этого класс становится доступен

**Пример использования:**

```parser3
# /project/auto.p
@autouse[name]
^use[/classes/$name.p]

# /project/classes/User.p
@CLASS
User

@create[name]
$self.name[$name]

# /project/pages/index.p
# User.p НЕ подключён явно, но класс доступен благодаря @autouse
@main[]
$user[^User::create[John]]  # @autouse автоматически подключит User.p
```

**Преимущества:**
- Не нужно явно подключать каждый файл с классом
- Классы загружаются только когда реально используются (lazy loading)
- Упрощает структуру зависимостей

**Важно для IDE:** Если виден `@autouse[]` — все классы проекта потенциально доступны.

### 5.6. Что становится доступным после подключения

- Все классы из подключённого файла
- Методы MAIN из подключённого файла

## 6. Таблица сравнения вызовов

| Синтаксис | Что ищет | Где ищет |
|-----------|----------|----------|
| `^method[]` | Метод | Текущий класс → @BASE цепочка → MAIN → use-файлы → auto.p |
| `^self.method[]` (в классе) | Метод объекта | Текущий класс → @BASE цепочка |
| `^self.method[]` (в MAIN) | Метод MAIN | MAIN текущего файла → MAIN use-файлов (= `^MAIN:method[]`) |
| `^MAIN:method[]` | Метод MAIN | MAIN текущего файла → MAIN use-файлов |
| `^BASE:method[]` | Метод базового класса | @BASE класс → его @BASE цепочка |
| `^ClassName:method[]` | Статический метод | Указанный класс → его @BASE цепочка |
| `^ClassName::create[]` | Конструктор | Указанный класс |

## 7. Валидные и невалидные форматы вызова

Этот раздел описывает какие форматы вызова методов корректны в Parser3.

### 7.1. В классе MAIN (код вне @CLASS)

| Формат | Валидность | Описание |
|--------|------------|----------|
| `^method[]` | ✅ Валидно | Вызов метода из MAIN или подключённых файлов |
| `^self.method[]` | ✅ Валидно | Эквивалентно `^MAIN:method[]` |
| `^MAIN:method[]` | ✅ Валидно | Явный вызов метода MAIN |
| `^ClassName::create[]` | ✅ Валидно | Создание объекта класса |
| `^ClassName:method[]` | ✅ Валидно | Статический вызов метода класса |
| `^BASE:method[]` | ❌ Невалидно | MAIN не может иметь @BASE |
| `^method.something[]` | ❌ Невалидно | Вызов метода объекта из переменной |
| `^self.method.something[]` | ❌ Невалидно | Точка после имени метода |

### 7.2. В именованном классе (внутри @CLASS)

| Формат | Валидность | Описание |
|--------|------------|----------|
| `^method[]` | ✅ Валидно | Вызов метода текущего класса, @BASE или MAIN |
| `^self.method[]` | ✅ Валидно | Вызов метода текущего класса или @BASE (НЕ ищет в MAIN) |
| `^MAIN:method[]` | ✅ Валидно | Явный вызов метода MAIN |
| `^BASE:method[]` | ✅ Валидно* | Вызов метода базового класса (*только если есть @BASE) |
| `^BASE::method[]` | ✅ Валидно* | То же, двойное двоеточие допустимо |
| `^ClassName::create[]` | ✅ Валидно | Создание объекта класса |
| `^ClassName:method[]` | ✅ Валидно | Статический вызов метода класса |
| `^method.something[]` | ❌ Невалидно | Вызов метода объекта из переменной |
| `^ClassName::method.something[]` | ❌ Невалидно | Точка после имени метода |
| `^self.method.something[]` | ❌ Невалидно | Точка после имени метода |
| `^BASE:method.something[]` | ❌ Невалидно | Точка после имени метода |

### 7.3. Формат `^method.something[]` — вызов метода объекта

Этот формат означает вызов метода `something` у **объекта**, который возвращает `^method[]`:

```parser3
# Пример валидного использования в runtime
$obj[^User::create[]]
^obj.validate[]        # Вызов метода validate у объекта $obj

# Или в одну строку
^User::create[].validate[]  # Создать объект и сразу вызвать его метод
```

**Почему IDE не резолвит `^method.something[]`:**
- `^method` возвращает динамический объект
- Тип объекта неизвестен на этапе статического анализа
- Метод `something` может быть у любого класса

### 7.4. Сводная таблица валидности

| Контекст | `^m[]` | `^self.m[]` | `^MAIN:m[]` | `^BASE:m[]` | `^Class:m[]` | `^Class::m[]` | `^m.x[]` |
|----------|--------|-------------|-------------|-------------|--------------|---------------|----------|
| MAIN | ✅ | ✅* | ✅ | ❌ | ✅ | ✅ | ❌ |
| @CLASS без @BASE | ✅ | ✅ | ✅ | ❌ | ✅ | ✅ | ❌ |
| @CLASS с @BASE | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ❌ |

\* В MAIN `^self.m[]` эквивалентен `^MAIN:m[]`

## 8. Примеры сложных случаев

### 8.1. Переопределение метода с вызовом родительского

```parser3
@CLASS
User

@validate[]
^if(!$self.email){
    ^throw[error;Email required]
}

@CLASS
Admin

@BASE
User

@validate[]
^BASE:validate[]  # Сначала вызываем родительскую валидацию
^if(!$self.role){
    ^throw[error;Role required]
}
```

### 8.2. Вызов метода из другого класса

```parser3
@CLASS
Logger

@log[message]
^console:line[$message]

@CLASS
User

@create[name]
$self.name[$name]
^Logger:log[User created: $name]  # Статический вызов метода Logger
```

### 8.3. Разрешение конфликтов имён

```parser3
@main[]
^process[]  # Вызовет MAIN:process или метод класса?

@process[]
# Это MAIN:process

@CLASS
Worker

@run[]
^process[]        # Ищет в Worker, потом в MAIN
^self.process[]   # Только в Worker (НЕ в MAIN!)
^MAIN:process[]   # Только в MAIN

@process[]
# Это Worker.process
```

### 8.4. self в MAIN vs self в классе

```parser3
# auto.p
@method[]
# Метод в MAIN

# index.p
@main[]
^self.method[]    # Найдёт @method из auto.p (self = MAIN)
^MAIN:method[]    # То же самое

@CLASS
User

@run[]
^self.method[]    # НЕ найдёт! self в классе — это только User
^MAIN:method[]    # Найдёт @method из auto.p
^method[]         # Найдёт — простой вызов ищет и в MAIN
```

### 8.5. Использование auto.p для общих зависимостей

```parser3
# /project/auto.p
^use[/lib/database.p]
^use[/lib/utils.p]

@connect[]
^mysql:connect[...]

# /project/pages/index.p
# Автоматически имеет доступ к ^connect[] из auto.p
@main[]
^connect[]
^render_page[]
```

### 8.6. Partial-классы

```parser3
# User.p
@CLASS
User

@OPTIONS
partial

@create[name]
$self.name[$name]

# UserAuth.p  
@CLASS
User

@OPTIONS
partial

@login[password]
^if($password eq $self.password){...}

# index.p
@main[]
$user[^User::create[John]]
^user.login[secret]  # Метод из UserAuth.p
```

### 8.7. Полный пример с валидными и невалидными вызовами

```parser3
# Файл: /project/pages/index.p

@main[]
^connect[]              # ✅ Ищет в MAIN
^self.connect[]         # ✅ Ищет в MAIN (self = MAIN)
^MAIN:connect[]         # ✅ Явно в MAIN
^User::create[]         # ✅ Конструктор класса User
^User:validate[$email]  # ✅ Статический метод User
^BASE:something[]       # ❌ MAIN не имеет @BASE
^connect.close[]        # ❌ Вызов метода объекта

@connect[]
$db[...]

@CLASS
User

@BASE
BaseEntity

@create[name]
$self.name[$name]
^self.init[]            # ✅ Ищет в User, потом в BaseEntity (НЕ в MAIN!)
^BASE:init[]            # ✅ Ищет в BaseEntity
^MAIN:connect[]         # ✅ Ищет в MAIN
^validate[]             # ✅ Ищет в User → BaseEntity → MAIN
^self.validate.check[]  # ❌ Точка после метода

@validate[]
^if(!$self.name){
    ^throw[error]
}

@CLASS  
Admin

@BASE
User

@create[name;role]
^BASE:create[$name]     # ✅ Вызывает User:create
$self.role[$role]

@ban[user_id]
^User:find[$user_id]    # ✅ Статический вызов
^self.log[ban]          # ✅ Ищет в Admin → User → BaseEntity
^User::create.init[]    # ❌ Точка после create
```

## 9. Служебные символы и экранирование

### 9.1. Служебные символы Parser3

Следующие символы являются служебными в Parser3:

```
^   $   ;   @   (   )   [   ]   {   }   "   :   #
```

### 9.2. Экранирование символом ^

Чтобы отменить специальное действие служебных символов, их необходимо предварять символом `^`:

| Запись | Результат | Описание |
|--------|-----------|----------|
| `^$` | `$` | Литерал доллара, не переменная |
| `^^` | `^` | Литерал каретки, не начало метода |
| `^{` | `{` | Литерал скобки, не открытие блока |
| `^}` | `}` | Литерал скобки, не закрытие блока |
| `^[` | `[` | Литерал скобки |
| `^]` | `]` | Литерал скобки |
| `^(` | `(` | Литерал скобки |
| `^)` | `)` | Литерал скобки |
| `^;` | `;` | Литерал точки с запятой, не разделитель параметров |
| `^"` | `"` | Литерал кавычки |
| `^:` | `:` | Литерал двоеточия |
| `^#` | `#` | Литерал решётки (но см. hex-коды ниже) |
| `^#XX` | символ | Символ по hex-коду (например `^#20` — пробел) |

Parser3 распознаёт hex-код только если после `^#` идут ровно две hex-цифры (`0-9`, `A-F`, `a-f`).
Если двух hex-цифр нет, `^#` остаётся обычным экранированием решётки, а следующие символы разбираются как текст.
`^#00` в оригинальном Parser3 запрещён и даёт `BAD_HEX_LITERAL`; подсветка плагина сейчас только выделяет `^#HH` как escape-последовательность и не делает отдельную диагностику этой ошибки.

**ВАЖНО:** `^` экранирует ТОЛЬКО служебные символы! Если после `^` идёт буква или `_` — это **вызов метода**, а не экранирование:

| Запись | Интерпретация |
|--------|---------------|
| `^)` | Экранированная `)` — литерал |
| `^method` | Вызов метода `method` |
| `^m` | Вызов метода `m` |
| `^_private` | Вызов метода `_private` |

```parser3
# Примеры в строках внутри условий
^if($a eq '^)'){...}        # Сравнение с литералом ")"
^if($a eq '^method[]'){...} # Сравнение с результатом вызова ^method[]

# То же самое с двойными кавычками
^if($a eq "x^)"){...}       # Сравнение с "x)" 
^if($a eq "^method[]"){...} # Сравнение с результатом ^method[]
```

### 9.3. Обязательное vs контекстное экранирование

**Всегда нужно экранировать:**
- `^$` — символ `$` всегда начинает переменную
- `^^` — символ `^` всегда начинает метод или экранирование

**Контекстно-зависимое экранирование:**

| Символ | Когда нужно экранировать |
|--------|--------------------------|
| `;` | Внутри параметров методов (разделитель параметров) |
| `(`, `)` | Когда могут быть восприняты как конструкция языка |
| `{`, `}` | Непарные скобки внутри параметров |
| `[`, `]` | Непарные скобки внутри параметров |
| `"` | Внутри строк |
| `:` | Когда может быть воспринято как разделитель |

### 9.4. Примеры экранирования

```parser3
# Вывод цены с долларом
Цена: ^$100

# Вывод символа каретки
Используйте ^^method для вызова

# Экранирование в параметрах метода
^method[текст с ^; точкой с запятой]

# Переменная с последующей скобкой
$var^(13^)  # Выводит значение $var и затем (13)
# Без экранирования $var(13) — это присвоение 13 в $var

# Непарная скобка в параметре
^method{текст с непарной ^} скобкой}

# Парные скобки можно не экранировать
^method[текст с парными скобками [ и ]]

# Hex-код символа
^#20  # пробел
^#0A  # перевод строки
^#09  # табуляция
^#A0  # nbsp
^#no  # не hex-код: ^# — литерал решётки, затем текст no
```

### 9.5. Принцип чётности при экранировании

Экранирование работает по принципу чётности — каждый `^` экранирует следующий символ:

| Запись | Результат | Объяснение |
|--------|-----------|------------|
| `^}` | `}` | Экранированная скобка |
| `^^}` | `^` + конец блока | Экранированный `^`, затем закрывающая скобка |
| `^^^}` | `^}` | Экранированный `^` + экранированная `}` |
| `^^^^}` | `^^` + конец блока | Два экранированных `^`, затем закрывающая скобка |
| `^^^^^}` | `^^}` | Два экранированных `^` + экранированная `}` |

**Правило:** Нечётное количество `^` перед символом — символ экранирован. Чётное — символ имеет своё обычное значение.

```parser3
# Примеры в блоках
^method{
    текст ^}         # выводит: текст }
    текст ^^}        # КОНЕЦ БЛОКА, выводит: текст ^
}

^method{
    текст ^^^}       # выводит: текст ^}
    текст ^^^^}      # КОНЕЦ БЛОКА, выводит: текст ^^
}
```

### 9.6. Экранирование внутри SQL-блоков

Внутри SQL-блоков (`^table::sql{...}`, `^void:sql{...}` и т.д.) действуют те же правила экранирования:

```parser3
^void:sql{
    # ^$ — это литерал $, не переменная Parser3
    SELECT * FROM products WHERE price > '^$100'
    
    # Переменная Parser3 внутри SQL
    SELECT * FROM $table_name WHERE id = $id
    
    # Экранированная закрывающая скобка
    UPDATE t SET data = 'value with ^} brace'
    
    # Принцип чётности работает и здесь:
    # ^^} — это конец SQL-блока
    # ^^^} — выводит ^}
}
```

**Важно для лексера:**
- Экранирование `^` работает **везде** — и вне строк, и внутри строк `'...'` и `"..."`
- `^)` внутри `"x^)"` — это экранированная `)`, строка содержит `x)`
- `^method[]` внутри `'^method[]'` — это **вызов метода**, результат которого попадёт в строку
- SQL-строки `'...'` внутри SQL-блоков — это SQL-литералы, но Parser3-конструкции (`^method[]`, `$var`) внутри них **работают**

## 10. Локальные переменные

### 10.1. Общие правила

В Parser3 по умолчанию все переменные **глобальные**. Локальные переменные создаются через:
1. Директиву `@OPTIONS locals`
2. Модификатор `[locals]` при объявлении метода
3. Явное перечисление локальных переменных `[x;y]`

**Важно:** имя переменной тоже может начинаться с цифры. Например, `$2var[...]` — валидная переменная и должна резолвиться так же, как `$var[...]`.

### 10.2. @OPTIONS locals в MAIN

```parser3
@OPTIONS
locals

@helper1[]
$x[1]  # локальная переменная

@helper2[]
$y[2]  # локальная переменная
```

**Действие:** С момента встречи `@OPTIONS locals` и до конца MAIN класса все переменные в методах MAIN будут локальными.

**Не влияет на классы:** Если после `@OPTIONS locals` в MAIN идёт `@CLASS`, то внутри класса переменные глобальные (если у класса нет своей `@OPTIONS locals`).

### 10.3. @OPTIONS locals в классе

```parser3
@CLASS
Test

@OPTIONS
locals

@method1[]
$x[1]  # локальная переменная

@method2[]
$y[2]  # локальная переменная
```

**Действие:** С момента встречи `@OPTIONS locals` внутри класса все переменные в методах этого класса будут локальными.

**Область действия:** Только текущий класс. Не влияет на MAIN и другие классы.

### 10.4. Модификатор [locals]

```parser3
@method[][locals]
$x[1]  # локальная
$y[2]  # локальная
```

**Действие:** Все переменные внутри этого метода будут локальными.

### 10.5. Явное перечисление локальных переменных

```parser3
@method[][x;y]
$x[1]  # локальная
$y[2]  # локальная
$z[3]  # глобальная
```

**Действие:** Только переменные `x` и `y` локальные, остальные глобальные.

### 10.6. Приоритет правил

1. **@OPTIONS locals в файле** — переопределяет всё, все переменные локальные независимо от `[locals]` или `[x;y]`
2. **[locals]** — все переменные метода локальные
3. **[x;y]** — только x и y локальные

```parser3
@OPTIONS
locals

@method[][x;y]  # Игнорируется! Все переменные всё равно локальные из-за @OPTIONS
$x[1]  # локальная
$y[2]  # локальная
$z[3]  # локальная (из-за @OPTIONS locals)
```

### 10.7. Глобальные переменные через $MAIN:var

```parser3
@OPTIONS
locals

@method[]
$x[1]           # локальная
$MAIN:g[2]      # глобальная переменная в MAIN
```

**В любом месте** (класс или MAIN) можно создать/прочитать глобальную переменную MAIN через `$MAIN:var`.

### 10.8. Переменные класса

#### Без @OPTIONS locals

```parser3
@CLASS
Test

@method[]
$x[5]        # глобальная переменная класса
$self.x[5]   # то же самое — глобальная переменная класса

@method2[]
^rem[Читаем x из method[]]
$result[$x]  # или $self.x — одно и то же
```

**$var и $self.var в классе БЕЗ locals — это одно и то же** (глобальная переменная класса).

#### С @OPTIONS locals

```parser3
@CLASS
Test

@OPTIONS
locals

@method[]
$x[5]        # локальная переменная метода
$self.x[6]   # глобальная переменная класса

@method2[]
$result[$self.x]  # 6 (глобальная переменная класса)
# $x здесь недоступна — она локальная в method[]
```

**$self.var всегда глобальная** переменная класса (даже с locals).

### 10.9. Чтение переменных класса из MAIN

```parser3
@CLASS
Test

@method[]
$x[5]        # глобальная переменная класса
$self.x[5]   # то же самое

# В MAIN
@main[]
$val[$Test:x]  # Читаем переменную класса Test
```

**Из MAIN:** Переменные класса читаются через `$ClassName:var`.

### 10.10. Запись переменных класса из другого места

```parser3
# В MAIN или другом классе
@main[]
$x[$Test:var]       # Читаем переменную класса Test
$Test:var2[value]   # Записываем в переменную класса Test

# Класс
@CLASS
Test

@create[]
$var[x]
```

### 10.11. ^use[] и @OPTIONS locals

```parser3
# file1.p
@helper1[]
$x[1]  # глобальная

^use[file2.p]

@helper2[]
$y[2]  # локальная (из-за @OPTIONS locals в file2.p)

# file2.p
@OPTIONS
locals
```

**^use[] включает код файла** — если в подключаемом файле есть `@OPTIONS locals` в MAIN, то все методы в текущем файле ПОСЛЕ `^use[]` будут с локальными переменными.

**Для плагина:** Идём по файлу сверху вниз, фиксируем все `^use[]` независимо от условий (`^if`, `^switch` и т.д.).

### 10.12. Partial классы и @OPTIONS locals

```parser3
# User.p
@CLASS
User

@OPTIONS
partial
locals

@method1[]
$x[1]  # локальная

# UserAuth.p
@CLASS
User

@OPTIONS
partial
# НЕТ locals

@method2[]
$y[2]  # локальная (из-за locals в User.p выше)
```

**Порядок подключения имеет значение:**
- Если файл с `locals` подключается первым — методы ниже будут с локальными переменными
- `@OPTIONS locals` действует на методы **ниже** в порядке подключения файлов

### 10.13. Примеры использования локальных переменных

#### Пример 1: Класс с locals

```parser3
@CLASS
Test

@OPTIONS
locals

@method[]
$var[x]        # локальная переменная метода
$self.var[y]   # глобальная переменная класса

@method2[]
$result[$self.var]  # y

# В файле, который вызывает:
^test:method[]
^test:method2[]  # выведет "result=y"
```

#### Пример 2: Глобальная переменная MAIN в классе

```parser3
@CLASS
Test

@OPTIONS
locals

@method[]
$MAIN:g_var[z]  # глобальная переменная для MAIN

# В MAIN (вне класса)
@main[]
$x[$MAIN:g_var]  # z
$y[$g_var]       # z (то же самое)
```

#### Пример 3: Приоритет @OPTIONS

```parser3
@OPTIONS
locals

@method1[]
$x[1]  # локальная

@method2[][a;b]
$a[2]  # локальная (но из-за @OPTIONS, не из-за [a;b])
$b[3]  # локальная (но из-за @OPTIONS, не из-за [a;b])
$c[4]  # локальная (из-за @OPTIONS locals)

@method3[][locals]
$d[5]  # локальная (из-за @OPTIONS locals, не из-за [locals])
```

**@OPTIONS locals переопределяет всё** — модификаторы `[locals]` и `[a;b]` игнорируются.

### 10.14. Специальная переменная $result

`$result` — возвращаемое значение метода. **Всегда локальна в пределах текущего метода**, независимо от `@OPTIONS locals` или `[locals]`.

```parser3
@main[]
$result[x]       # возвращаемое значение @main — локальная

@method[]
$result[y]       # возвращаемое значение @method — другая переменная, тоже локальная
```

**Ключевые правила:**
- `$result` в `@main[]` и `$result` в `@method[]` — это **разные переменные**, не переопределение
- `$result` НЕ видна из других методов и файлов — её область видимости ограничена текущим методом
- Навигация и резолв `$result` работают только в пределах текущего метода
- `$result` не участвует в эквивалентности `$var` / `$self.var` / `$MAIN:var`
- `^return[...]` для типизации результата метода эквивалентен `$result[...]`
- Пустые формы `$result[]` и `^return[]` не задают новый тип результата и не затирают уже найденный непустой тип
- Если в методе есть несколько структурных присваиваний результата одного типа, итоговым считается поздняя структурная форма: она может удалить или переименовать ключи через `sub`/`rename`
- Если позднее присваивание слабое и не даёт структуры (`$result(1)`, простой скаляр), для IDE-типизации сохраняется самая богатая ранее найденная структура результата: hash/table/object не должен пропадать из автокомплита и навигации

```parser3
@getData[]
$result[^table::create{name	value}]
$result.name     # → навигация к $result[^table::create{...}] выше в том же методе

@process[]
$result[ok]      # другая переменная, никак не связана с $result в @getData
```

```parser3
@method[]
$result[^hash::create[]]
^return[]        # не сбрасывает тип результата

@method2[]
$result[$.x[$.y[z]]]
^return[$result] # для типизации эквивалентно новому присваиванию результата
```

## 11. Эквивалентные формы переменных в MAIN

### 11.1. $var, $self.var, $MAIN:var — одно и то же

В контексте MAIN класса следующие три формы обращения к переменной **абсолютно эквивалентны**:

```parser3
# В MAIN (вне @CLASS)
$var[^User::create[]]       # присваивание через $var
$self.var[^User::create[]]  # то же самое — self в MAIN = MAIN
$MAIN:var[^User::create[]]  # то же самое — явная ссылка на MAIN
```

Аналогично при чтении:
```parser3
@main[]
$a[$var]        # читаем var
$b[$self.var]   # то же самое
$c[$MAIN:var]   # то же самое
```

**Почему:** В MAIN контексте `self` — это сам MAIN, и `$MAIN:var` — явная ссылка на ту же переменную.

### 11.2. Вызов методов объекта — все формы эквивалентны

Если переменная содержит объект класса, вызов метода объекта через все три формы — одно и то же:

```parser3
# Все три варианта присваивания — одно и то же:
$user[^User::create[John]]
$self.user[^User::create[John]]
$MAIN:user[^User::create[John]]

# Все три варианта вызова метода объекта — одно и то же:
^user.validate[]
^self.user.validate[]
^MAIN:user.validate[]
```

### 11.3. Таблица эквивалентности

| Присваивание | Чтение | Вызов метода объекта |
|--------------|--------|----------------------|
| `$var[...]` | `$var` | `^var.method[]` |
| `$self.var[...]` | `$self.var` | `^self.var.method[]` |
| `$MAIN:var[...]` | `$MAIN:var` | `^MAIN:var.method[]` |

Все три строки — **одна и та же переменная**, один и тот же объект, один и тот же метод.

### 11.4. Важно: это только для MAIN

Эквивалентность `$var` и `$self.var` работает только в MAIN (или в классе без `@OPTIONS locals`). В классе с `@OPTIONS locals` это разные вещи:

```parser3
@CLASS
Test

@OPTIONS
locals

@method[]
$var[^User::create[]]       # ЛОКАЛЬНАЯ переменная метода
$self.var[^User::create[]]  # ГЛОБАЛЬНАЯ переменная класса — РАЗНЫЕ!

^var.validate[]              # метод объекта из локальной $var
^self.var.validate[]         # метод объекта из глобальной $self.var — РАЗНЫЕ!
``````

## 12. Синтаксис ${var} — переменная в фигурных скобках

### 12.1. Зачем нужен ${var}

Фигурные скобки позволяют явно обозначить границы имени переменной — когда после имени сразу идёт текст без пробела:

```parser3
# Без скобок — неоднозначно:
$prefixsuffix         # одна переменная "prefixsuffix"

# Со скобками — явно:
${prefix}suffix       # переменная "prefix", затем текст "suffix"
$prefix${suffix}      # текст "prefix", затем переменная "suffix"
```

### 12.2. Синтаксис цепочек ${data.key1.key2}

Внутри `${}` можно записать dotpath-цепочку — доступ к ключам хеша:

```parser3
${data}               # переменная data
${data.key1}          # ключ key1 хеша data
${data.key1.key2}     # вложенный ключ key2
```

Это эквивалентно `$data.key1.key2` — фигурные скобки лишь явно задают границы.

### 12.3. Представление в лексере плагина

Лексер разбивает `${data.key1.key2}` на отдельные токены (НЕ один токен):

| Текст | Токен |
|-------|-------|
| `$` | `DOLLAR_VARIABLE` |
| `{` | `LBRACE` |
| `data` | `VARIABLE` |
| `.` | `TEMPLATE_DATA` (точка) |
| `key1` | `VARIABLE` |
| `.` | `TEMPLATE_DATA` (точка) |
| `key2` | `VARIABLE` |
| `}` | `RBRACE` |

Это позволяет навигации (Ctrl+Click) и автокомплиту работать на каждом сегменте цепочки независимо.

### 12.4. Особые случаи

```parser3
${dynamic_name}       # простая переменная — $ + { + VARIABLE(dynamic_name) + }
$prefix${suffix}      # $prefix — одна переменная, ${suffix} — другая
```

`$prefix${suffix}` лексер разбивает как: `DOLLAR_VARIABLE` + `VARIABLE(prefix)` + `DOLLAR_VARIABLE` + `LBRACE` + `VARIABLE(suffix)` + `RBRACE`.
