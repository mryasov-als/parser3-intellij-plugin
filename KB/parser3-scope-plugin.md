# Область видимости в плагине IntelliJ IDEA для Parser3

## 1. Режимы автодополнения методов

Плагин поддерживает два режима работы автодополнения, настраиваемых в `Settings → Parser3`:

### 1.1. Режим "Все методы проекта" (по умолчанию)

**Поведение:** Автодополнение показывает все методы из всех `.p` файлов проекта.

**Когда полезен:**
- При рефакторинге — найти все методы с определённым именем
- При изучении кодовой базы
- Когда структура `@USE` ещё не настроена
- Для быстрого старта без дополнительной конфигурации

**Недостатки:**
- Показывает методы, которые могут быть недоступны в runtime
- Много вариантов в автодополнении

### 1.2. Режим "Только через @USE/^use[]"

**Поведение:** Автодополнение показывает только методы из файлов, которые явно подключены через `@USE`, `^use[]` или `auto.p`.

**Алгоритм построения области видимости:**

1. **Начальная точка** — текущий редактируемый файл
2. **Сбор auto.p файлов** — поиск всех `auto.p` по иерархии директорий (сверху вниз, от корня к файлу)
3. **Рекурсивный обход** всех директив `@USE` и вызовов `^use[]`:
   - Парсинг директив `@USE` в начале файла
   - Поиск вызовов `^use[path]` в коде
   - Разрешение путей относительно CLASS_PATH или корня проекта
4. **Построение графа зависимостей** — какие файлы подключают какие
5. **Результат** — список "видимых" файлов для текущего файла

**Пример:**

```
main.p
├── auto.p (цепочка от корня проекта к директории main.p)
├── @USE /classes/User.p
│   └── @USE /classes/BaseEntity.p
└── @USE /lib/utils.p
```

При редактировании `main.p` видны методы из:
- `main.p` (собственные)
- Всех `auto.p` по иерархии от корня к текущей директории
- `/classes/User.p`
- `/classes/BaseEntity.p` (транзитивная зависимость)
- `/lib/utils.p`

**Преимущества:**
- Показывает только реально доступные методы
- Меньше "шума" в автодополнении
- Соответствует реальной области видимости Parser3

## 2. Настройки проекта

### 2.1. Основной auto.p (Main auto.p)

**Расположение:** `Settings → Parser3 → Main auto.p file`

**Назначение:** Указывает конфигурационный файл `auto.p`, который выполняется самым первым при запуске Parser3.

**Как работает:**
- Этот файл подключается раньше всех остальных `auto.p`
- Обычно располагается рядом с `parser3.cgi` (например, `document_root/cgi-bin/auto.p`)
- Может находиться как внутри `document_root`, так и вне его
- Все `^use[]` и `@USE` из этого файла становятся доступны везде в проекте
- Плагин начинает построение области видимости именно с этого файла

**Пример:**
```
/var/www/cgi-bin/auto.p  ← Конфигурационный auto.p (указываем в настройках)
/var/www/html/           ← document_root
```

### 2.2. CLASS_PATH

**Расположение:** `Settings → Parser3 → CLASS_PATH`

**Назначение:** Базовый путь для разрешения относительных путей в `^use[]` и `@USE`.

**Как работает:**
- Пути вида `/classes/User.p` разрешаются относительно CLASS_PATH
- Если CLASS_PATH = `/project`, то `/classes/User.p` → `/project/classes/User.p`
- Позволяет использовать абсолютные пути в коде независимо от расположения файла
- **Не влияет** на автоматическое подключение файлов `auto.p`

**Пример:**
```
CLASS_PATH = /var/www/myproject

# В коде:
^use[/lib/utils.p]  
# Разрешается в: /var/www/myproject/lib/utils.p
```

### 2.3. Динамическое изменение CLASS_PATH в коде

CLASS_PATH может быть изменён или дополнен непосредственно в коде Parser3:

```parser3
# В auto.p или любом другом файле:
^CLASS_PATH.append{/var/www/shared-libs}
```

**Как плагин обрабатывает это:**

- Плагин парсит файлы и отслеживает изменения `$CLASS_PATH`
- Изменения CLASS_PATH влияют на разрешение путей в `^use[]` и `@USE` **ниже по коду**
- Это влияет на то, какие методы видны в текущем файле

**Пример:**

```parser3
# /project/auto.p
^use[/classes/User.p]  # → разрешается относительно начального CLASS_PATH

# Добавляем дополнительный путь
^CLASS_PATH.append{/var/www/shared}
^use[/lib/utils.p]     # → теперь ищет также в /var/www/shared
```

**Важно:**
- Порядок путей в CLASS_PATH имеет значение
- Изменения CLASS_PATH в одном файле влияют на видимость методов в файлах, которые подключаются после этого изменения
- Плагин учитывает эти изменения при построении области видимости

## 3. Обработка auto.p

### 3.1. Автоматическое обнаружение

Плагин автоматически находит файлы `auto.p` по иерархии директорий от корня проекта к директории текущего файла:

```
/project/
├── auto.p                    # (1) Включается первым
├── classes/
│   ├── auto.p                # (2) Включается вторым
│   └── user/
│       ├── auto.p            # (3) Включается третьим
│       └── User.p            # ← Текущий файл
```

### 3.2. Порядок обработки

1. **Основной auto.p из настроек** (конфигурационный файл) — выполняется самым первым
2. `auto.p` из корня проекта
3. `auto.p` из следующей директории по иерархии вниз
4. И так далее до директории текущего файла

**Порядок: сверху вниз** — от корня проекта к директории текущего файла.

### 3.3. Транзитивные зависимости

Все `^use[]` и `@USE` внутри `auto.p` также обрабатываются рекурсивно.

## 4. Метод @auto[]

### 4.1. Что такое @auto[]

Метод `@auto[]` — это инициализатор файла. Если в файле определён метод `@auto[]`, он автоматически выполняется при первом обращении к этому файлу.

```parser3
@auto[]
# Этот код выполнится автоматически при первом обращении к файлу
^connect_to_db[]
$initialized(true)

@process[]
...
```

### 4.2. Важно: @auto[] не влияет на видимость

**Метод `@auto[]` не влияет на подключение или видимость файла:**

- Файл с методом `@auto[]` всё равно должен быть явно подключён через `@USE`, `^use[]` или быть файлом с именем `auto.p`
- Метод `@auto[]` — это просто инициализатор, который выполняется автоматически при загрузке файла
- Плагин не делает никаких исключений для файлов с методом `@auto[]`

**Не путать:**
- **Файл `auto.p`** — подключается автоматически по иерархии директорий
- **Метод `@auto[]`** — выполняется автоматически при загрузке файла (но файл нужно подключить)

## 5. Метод @autouse[] — автозагрузка классов

### 5.1. Что такое @autouse[]

Метод `@autouse[]` позволяет автоматически загружать классы по требованию. Определяется в MAIN классе (обычно в корневом `auto.p`).

```parser3
# /project/auto.p
@autouse[name]
^use[/classes/$name.p]
```

### 5.2. Влияние на автодополнение классов в плагине

**Если из текущей позиции виден метод `@autouse[]` в MAIN классе:**

- Автодополнение **классов** работает так, как будто включён режим "Все методы проекта"
- Все классы из всех файлов проекта показываются в автодополнении
- Это позволяет писать `^User::create[]` без явного `^use[User.p]`

**Почему так:**
- `@autouse[]` загружает классы автоматически при первом обращении
- Поэтому любой класс проекта потенциально доступен
- Плагин показывает все классы, чтобы соответствовать реальному поведению Parser3

**Важно:** Это влияет только на автодополнение **классов** (после `^ClassName:`), не на методы.

### 5.3. Поиск @autouse[]

Плагин проверяет `@autouse[]` через позиционный `P3ScopeContext`: метод должен быть виден из текущего файла и текущего `cursorOffset`. `@autouse[]` ниже курсора в режиме "Только через @USE/^use[]" не расширяет видимость классов выше курсора.

Правило важно не смешивать с настройкой методов: `@autouse[]` расширяет поиск классов, но не делает видимыми обычные методы из всего проекта.

## 6. Контекстное автодополнение

Плагин анализирует контекст курсора и предлагает соответствующие варианты:

### 6.1. После `^` (начало вызова)

**Показывает:**
- Методы текущего класса
- Методы базовых классов (@BASE цепочка)
- Методы MAIN
- Конструкторы классов (`ClassName::create`)
- Статические методы классов (`ClassName:method`)
- Встроенные методы Parser3

### 6.2. После `^self.`

**Показывает только:**
- Методы текущего класса
- Методы базовых классов
- Переменные с известным типом (`varName.`) — в MAIN контексте `^self.` эквивалентен `^`, переменные те же что и после голого `^`

### 6.3. После `^MAIN:`

**Показывает только:**
- Методы MAIN из текущего файла
- Методы MAIN из подключённых файлов
- Переменные с известным типом (`varName.`) — в MAIN контексте `$MAIN:var` эквивалентен `$var`, переменные те же что и после голого `^`

### 6.4. После `^BASE:`

**Показывает только:**
- Методы базового класса (указанного в @BASE)
- Методы всей цепочки наследования базового класса

**Доступно только:** Внутри класса с объявленным `@BASE`

### 6.5. После `^ClassName:`

**Показывает:**
- Статические методы указанного класса
- Методы базовых классов указанного класса

### 6.6. После `^ClassName::`

**Показывает:**
- Конструкторы указанного класса (методы, создающие объект)

### 6.7. После `^var.` (метод объекта переменной)

Если переменная была присвоена результатом конструктора класса, плагин знает тип и показывает **методы и переменные** класса:

```parser3
$user[^User::create[]]
^user.          # автокомплит: методы User + переменные User (с точкой, для цепочки)
```

**Как работает:**
1. Плагин индексирует присваивания вида `$var[^ClassName::constructor[]]`
2. При вводе `^var.` находит тип переменной через индекс
3. Показывает методы класса `ClassName` (и его `@BASE` цепочки)
4. Показывает переменные класса (`$self.var[^Type::]`) с точкой для продолжения цепочки
5. Если тип неизвестен — показывает встроенные методы объектов

**Все три формы работают одинаково** (см. раздел 11 в parser3-scope-language.md):

```parser3
$user[^User::create[]]          # или $self.user[...] или $MAIN:user[...]
^user.validate[]                # ✅ автокомплит методов User
^self.user.validate[]           # ✅ то же самое
^MAIN:user.validate[]           # ✅ то же самое
```

**Реализация:**
- Индекс: `P3VariableTypeFileIndex` — индексирует присваивания `$var[^Class::]`, `$self.var[^Class::]`, `$MAIN:var[^Class::]`
- Сервис: `P3VariableIndex` — обёртка для поиска типа по имени переменной
- Автокомплит: `P3VariableMethodCompletionContributor.addClassMethods()` — методы + переменные с точкой
- Координация: `P3MethodCompletionContributor` — срезает `self.`/`MAIN:` из имени и вызывает поиск типа
- **UNKNOWN_TYPE fallback**: если тип переменной неизвестен (пустая строка `""`), показываются все встроенные методы объектов через `addBuiltinObjectMethods()`

### 6.8. После `$var.` (свойства объекта переменной)

В контексте `$var.` показываются **только свойства** — переменные класса и @GET_ геттеры, **без обычных методов**:

```parser3
$user[^User::create[]]
$user.          # автокомплит: свойства User (переменные + @GET_), БЕЗ точки
```

**Различие `^var.` vs `$var.`:**

| Контекст | Что показывается | Вставка |
|----------|-----------------|---------|
| `^user.` | Методы + переменные | `varName.` (с точкой) |
| `$user.` | Только свойства (@GET_ + $self.var) | `varName` (без точки) |

**@GET_ свойства:**

`@GET_xxx[]` в Parser3 создаёт свойство, доступное через `$obj.xxx`. При индексации `@GET_xxx` сохраняется как метод `xxx` с флагом `isGetter=true`. В `$var.` контексте показываются только методы с `isGetter=true`.

**Фильтрация переменных класса:**

Глобальные переменные класса (`$self.var[^Type::]`) показываются как свойства. Простые `$var` показываются только если:
1. Нет `@OPTIONS locals` у класса
2. Нет `[locals]` у метода
3. Переменная не в списке `[var1;var2]` у метода

**Реализация:**
- Автокомплит: `P3VariableMethodCompletionContributor.addClassProperties()` — свойства без точки
- Координация: `P3VariableCompletionContributor` — определяет contextType="objectMethod"

## 7. Индексация проекта

### 7.1. Что индексируется

- **Классы:** Имя, файл, позиция, @BASE, диапазон (startOffset, endOffset)
- **Методы:** Имя, параметры, класс-владелец, тип (конструктор/статический/объектный)
- **Зависимости:** Граф `@USE` и `^use[]` между файлами
- **auto.p файлы:** Расположение и содержимое
- **Типы переменных:** Присваивания `$var[^ClassName::...]` — ключ: имя переменной, значение: имя класса. Также матчит `$self.var[...]` и `$MAIN:var[...]` — все индексируются под чистым именем `var`

### 7.2. Когда обновляется индекс

- При открытии проекта
- При сохранении файла
- При изменении файла в редакторе (с задержкой)
- При добавлении/удалении файлов

### 7.3. Версионирование индекса

**Важно:** При изменении логики индексации нужно инкрементировать версию в `getVersion()`:

```java
// P3ClassFileIndex.java, P3MethodFileIndex.java, P3UseFileIndex.java
public int getVersion() {
    return 6; // Увеличить при изменении!
}
```

Без этого IDE будет использовать старый кеш и индекс не обновится.

## 8. Определение текущего класса

Для корректной работы `^self.`, `^BASE:` и контекстного автодополнения плагин определяет текущий класс по позиции курсора:

### 8.1. Алгоритм

1. Получаем все классы файла через `P3ClassIndex.findInFile()`
2. Для каждого класса проверяем диапазон `[startOffset, endOffset)`
3. Если позиция попадает в диапазон — это текущий класс
4. Если не попадает ни в один — это MAIN класс

### 8.2. Единственный источник: `P3ClassIndex.findClassAtOffset()`

```java
P3ClassIndex classIndex = P3ClassIndex.getInstance(project);
P3ClassDeclaration classDecl = classIndex.findClassAtOffset(file, offset);
String className = classDecl != null ? classDecl.getName() : "MAIN";
```

## 9. Разрешение @BASE

При автодополнении `^BASE:` или построении иерархии:

1. Читает имя базового класса из `@BASE` текущего класса
2. Ищет класс с таким именем в видимых файлах
3. Если найдено несколько — берёт последний (Parser3 семантика)
4. Рекурсивно строит цепочку наследования

## 10. Фильтрация методов по контексту класса

### 10.1. Простой вызов `^method[]`

При вызове `^method[]` ищутся методы:

**В текущем файле:**
- Только методы того же класса, в котором находится вызов

**В других файлах (через USE):**
- Только методы MAIN класса

**Пример:**

```parser3
@method[]           # MAIN:method
@main[]
^method[]           # → Найдёт только MAIN:method, НЕ Admin:method

@CLASS
Admin
@method[]           # Admin:method
```

### 10.2. Вызов `^ClassName::method[]`

Ищутся методы только в указанном классе (и его @BASE цепочке).

Если метод не найден в классе — навигация не работает (не fallback на класс).

## 11. Навигация (Ctrl+Click / Go to Definition)

### 11.1. Валидные форматы для навигации

| Формат | Куда переходит |
|--------|----------------|
| `^method[]` | К `@method[]` в текущем классе или MAIN |
| `^self.method[]` | К `@method[]` в текущем классе |
| `^MAIN:method[]` | К `@method[]` в MAIN |
| `^BASE:method[]` | К `@method[]` в базовом классе |
| `^ClassName:method[]` | К `@method[]` в указанном классе |
| `^ClassName::create[]` | К `@create[]` в указанном классе |
| `^var.method[]` | К `@method[]` в классе переменной (тип из `$var[^Class::]`) |
| `$var` | К определению переменной (`$var[...]`) |
| `$self.var` | К определению переменной (`$self.var[...]`) |
| `$MAIN:var` | К определению переменной (`$MAIN:var[...]`) |
| `$result` | К определению переменной (`$result[...]`) |

### 11.2. Навигация `^var.method[]`

При клике на `method` в `^var.method[]`:
1. Определяется тип переменной через `P3VariableIndex.resolveChainedType()`
2. Метод ищется в классе переменной и его `@BASE` цепочке
3. Метод **не ищется** в других классах — только в классе переменной

**ВАЖНО**: `P3MethodCallElement.getReferences()` возвращает `EMPTY_ARRAY` для `objectMethodCall`. Без этого `P3MethodReference.multiResolve()` находил бы метод в любом классе, игнорируя тип переменной.

### 11.3. Навигация `$var`

При клике на `$var`, `$self.var`, `$MAIN:var` или `$result`:
- Переход к первому присваиванию переменной
- Поиск через `P3VariableIndex.findVariableDefinitionLocation()`
- Те же правила видимости (эквивалентность `$var` = `$self.var` = `$MAIN:var` в MAIN)
- **$result**: токен IMPORTANT_VARIABLE, **всегда локальна в пределах текущего метода**. `$result` в `@main[]` и `$result` в `@method[]` — разные переменные. Навигация и резолв `$result` ищут определение только внутри текущего метода, не в других методах и не в других файлах.
- **Навигация на себя**: клик на определении не навигирует, fallback на `navigateToMethodParameter()`

### 11.4. Затенение параметрами метода

Параметр метода затеняет одноимённую внешнюю переменную:

```parser3
$var[x]           # MAIN

@m[var]           # параметр "var"
$var              # → навигация к параметру @m[var], НЕ к $var[x] в MAIN
$var[y]           # переопределение внутри метода
$var              # → навигация к $var[y] (присваивание в текущем методе)
```

**Правила приоритета:**
1. Присваивание в текущем методе → приоритет над параметром
2. Параметр → приоритет над переменными из MAIN/других методов/файлов
3. Только для `$var` (без prefix) — `$self.var` и `$MAIN:var` не затеняются

### 11.5. Невалидные форматы (навигация не работает)

| Формат | Почему невалиден |
|--------|------------------|
| `^ClassName::method.something[]` | Точка после имени метода |

### 11.6. Единый парсер: `P3PsiExtractor.parseMethodCall()`

Все компоненты навигации используют одну функцию для определения валидности:

```java
P3PsiExtractor.MethodCallInfo callInfo = P3PsiExtractor.parseMethodCall(element);
if (!callInfo.isValid()) {
        return PsiElement.EMPTY_ARRAY;
}
```

## 12. Popup при множественных результатах

### 12.1. Когда показывается popup

- Найдено несколько классов с одинаковым именем в разных файлах
- Найдено несколько методов с одинаковым именем

### 12.2. Отображение в popup

**Для классов:**
- Имя класса (не "@CLASS")
- Путь к файлу от document_root или от корня проекта

**Для методов:**
- Имя метода
- Путь к файлу

## 13. Таблица сравнения вызовов

| Синтаксис | Какие методы показываются |
|-----------|---------------------------|
| `^method[]` | Методы текущего класса + MAIN из USE-файлов |
| `^self.method[]` | Только объектные методы текущего класса |
| `^ClassName:method[]` | Статические методы указанного класса |
| `^ClassName::method[]` | Конструкторы указанного класса |

## 14. Приоритеты в автодополнении

Варианты сортируются по релевантности:

1. **Высший приоритет:** Методы текущего класса
2. **Высокий:** Методы базовых классов
3. **Средний:** Методы MAIN текущего файла
4. **Обычный:** Методы из подключённых файлов
5. **Низший:** Встроенные методы Parser3

## 15. Навигация по CSS классам

Плагин поддерживает навигацию от `class="className"` в HTML к определениям `.className` в CSS внутри Parser3-файлов.

### 15.1. Архитектура

Навигация реализована через несколько механизмов:

| Компонент | Назначение |
|-----------|------------|
| `Parser3HtmlInjector` | Инжектирует HTML в Parser3 файлы с `<style>` содержимым |
| `Parser3CssInjector` | Инжектирует CSS из `<style>` блоков |
| `Parser3CssClassReferenceContributor` | Добавляет references к CSS классам в injected HTML |
| `P3CssClassFileIndex` | Индексирует CSS классы из `<style>` блоков Parser3-файлов |
| `P3CssClassIndex` | Читает только `P3CssClassFileIndex` и находит определения в Parser3-файлах |

**Важно:** внешний `css/scss/less` больше не сканируется самописным кодом плагина. Эти файлы резолвятся штатной HTML/CSS инфраструктурой IDE. Старый глобальный обход стилевых файлов был убран, потому что вызывал тяжёлые фризы на больших проектах.

### 15.2. Как работает навигация (Ctrl+Click на class="item")

1. **Текущий Parser3 файл** — ищется `.item` в `<style>` блоках
2. **Другие Parser3 файлы** — ищется `.item` в `<style>` блоках других `.p` файлов через `P3CssClassFileIndex`
3. **Внешние CSS/SCSS/LESS файлы** — обрабатываются встроенной поддержкой IDE, не нашим индексом

При множественных результатах показывается popup с выбором.

### 15.3. Подсветка unused CSS классов

Для навигации и автокомплита внутри Parser3 используются references и индекс `P3CssClassFileIndex`.
Подсветка `unused` больше не опирается на старую PolySymbols-схему из этого раздела.

### 15.4. HTML Injection

HTML код в Parser3 файлах инжектируется как единый виртуальный документ:

```html
<style>
    .item { color: red; }
</style>
<div class="item">...</div>
```

**Важно:** CSS внутри `<style>` передаётся в HTML injection полностью — это необходимо, чтобы локальные классы из Parser3-файла были доступны для навигации и автокомплита.

### 15.5. Ключевые файлы

```
index/
├── P3CssClassFileIndex.java              # Индекс локальных CSS классов из Parser3
├── P3CssClassIndex.java                  # Поиск определений по индексу

resolve/
├── Parser3CssClassReferenceContributor.java  # References для навигации и completion

injector/
├── Parser3HtmlInjector.java              # HTML injection
├── Parser3CssInjector.java               # CSS injection
├── InjectorUtils.java                    # Общие утилиты
```

### 15.6. Известные ограничения

- Навигация работает только для статических CSS классов (не для динамически генерируемых)
- При множественных определениях в разных файлах показывается popup
- Подсветка unused работает только внутри одного Parser3 файла (не кросс-файловая)
